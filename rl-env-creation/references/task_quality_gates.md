# Task Quality Gates

Use these gates before training or claiming an environment is evaluation-ready.

## Required Metadata
- `task_id`
- `split`: train, dev, eval, private_eval, refresh
- `task_family`
- `capability_axis`
- `difficulty_band`
- `horizon` or expected turns
- `tool_count` and state-dependency count
- `verifier_type`
- `source` or provenance
- `construction_mode`
- `public_status`: public, private, post-cutoff, refreshed
- `reward_hack_controls`

## Admission Gates
- Solvability: reference solution, executable solution, expert solution, or calibrated model-panel evidence.
- Independent admission: task generator, solver, verifier, and reviewer should not all be the same prompt/model path. Use a separate generator family, executable oracle, expert review, adversarial review, or held-out human check for high-value task families.
- Unambiguity: one intended objective; explicit handling for alternate valid solutions.
- Verifier coverage: every rubric criterion or executable check maps to task intent.
- State validity: starting state, transitions, and terminal conditions are possible.
- Difficulty calibration: report performance by band; do not rely on aggregate mean score.
- Diversity: no dominant family/source/verifier/difficulty band unless intentionally scoped.
- Split hygiene: no seed, answer, hidden-test, verifier, or metadata leakage across train/dev/eval.
- Artifact coverage: PDFs, spreadsheets, slides, code, databases, and UI state need artifact-aware verifiers.
- Repeated-run stability: track pass@k, variance, and prompt/tool perturbation robustness.

## Difficulty Axes
- Horizon length.
- Tool count and unique tool count.
- State dependencies and hidden-state dependencies.
- Number of evidence surfaces.
- Decoy/shortcut pressure.
- Branching, loops, or conditional logic.
- Domain-specific rule burden.
- Need for recovery after failed tool calls.

## Minimum Reports
- Count and percentage by split, family, capability axis, difficulty band, verifier type, construction mode, and source.
- Saturation check: which bands are too easy or too hard.
- Solvability evidence type per task.
- Ambiguity and alternate-answer handling.
- Held-out/private set status.

## Blockers
- Task was generated by an LLM and only solved/verified by the same prompt path.
- Private policy, rubric, answer, hidden-test, grader, or metadata files are visible to the agent.
- Verifier checks formatting while the target competency is reasoning, state mutation, or artifact correctness.
- A public static benchmark is used as a training gate without contamination controls.
- Hard tasks and easy saturated tasks are mixed into one mean score.
- Unsafe successful actions are counted as clean success.
